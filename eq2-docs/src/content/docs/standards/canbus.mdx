**<u>CAN Standards v3.0.2</u>**

CANBus messages will not have responses, only requests.
## CAN Bus Standards
### Header
The header specifies three key segments:

| GroupID | Device | Command ID |
| ------- | ------ | ---------- |
| 2 bits  | 5 bits |   4 bits   |

#### GroupID
Specifies the network segment of devices (group) intended to receive the message. A group typically has a size of 32 devices. There are 4 possible groups within the network.

Due to collisions with MyActuator namespace, group 0x1 is unusable. Only Groups 0x00, 0x02 and 0x03 are available.

Group allocations are:

| Group ID | Devices  |
| -------- | -------- |
|   0x00   | Onboard  |
|   0x01   | Reserved |
|   0x02   | Payload  |
|   0x03   | Unused   |


#### Device
Specifies the device within the group that is sending or receiving the message.

#### Command ID
Indicates the type of message being sent and the intended outcome on the machine.

These are:

|  ID  |  Command  | Sent From | Unit Size (bits) | Description |
| ---- | --------- | --------- | ---------------- | ----------- |
| 0x00 | Ping      | Either    | 0  | Checks whether a device is connected, also used as heartbeat payload |
| 0x01 | EStop     | Master    | 0  | Stops the device completely |
| 0x02 | TxInt8    | Master    | 8  | Sends an integer value |
| 0x03 | TxInt16   | Master    | 16 | Sends an integer value |
| 0x04 | TxFloat   | Master    | 32 | Sends a floating point value |
| 0x05 | TxData    | Master    | 64 | Sends an arbitrary byte buffer |
| 0x06 | TxData2   | Master    | 64 | Sends an arbitrary byte buffer |
| 0x0A | BrInt8    | Slave     | 8  | Receives an integer value |
| 0x0B | BrInt16   | Slave     | 16 | Receives an integer value |
| 0x0C | BrFloat   | Slave     | 32 | Receives a floating point value |
| 0x0D | BrData    | Slave     | 64 | Broadcasts an arbitrary byte buffer |
| 0x0E | BrData2   | Slave     | 64 | Broadcasts an arbitrary byte buffer |
 
Each packet will include between 1 and 'n' inclusions of the datatype in little endian order.

For example, a TxInt16 has a size of 16 bits. This means that up to 4 signed integers can be packed in the packet as follows:

| Offset |     Byte 1      |      Byte 2     |
| ------ | --------------- | --------------- |
| 0x00   | A (Int16)[0]    | A (Int16)[1]    | 
| 0x02   | B (Int16)[0]    | B (Int16)[1]    |
| 0x04   | C (Int16)[0]    | C (Int16)[1]    |
| 0x06   | D (Int16)[0]    | D (Int16)[1]    |

Messages are intended to be asynchonous and not involve transactions. This requires a listening thread to handle broadcast messages separately to the sending thread.

### PDB

- Firmware will broadcast status messages every 1 second by default.
- Each channel will present with a seperate device id. For example, a switchboard with 8 channels may occupy the device ids of 0 to 7. 
- It is recommended to use the upper bits of the device id to specify which switchboard is selected. In this scenario, the device id netmask would be 0x1c.

Device ID Sub-Structure:
[ID Segment - 2 bits][Channel ID - 3 bits]

<Aside>
    In the **message headers**, the "Board" section of the table is not actually part of the message header, but instead states what each ID Segment and Channel ID associate with on the board.
</Aside>
